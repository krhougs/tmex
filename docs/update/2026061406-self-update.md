# 程序内自更新与版本展示

## 背景

tmex 通过 npm 包 `tmex-cli`（`packages/app`）发布。此前更新只能在终端执行 `npx tmex-cli@<version> upgrade`，前端无版本展示、无更新入口。本特性补齐：版本展示、安装方式检测、程序内自更新（带状态机）、发版自动生成 changelog。

## 版本真相源与注入

「monorepo 版本」= 发布的 `tmex-cli` 版本（`packages/app/package.json.version`），唯一真相源。

纯函数 `formatDisplayVersion(base, isProd)`（`@tmex/shared`，浏览器安全）统一展示规则：`isProd ? base : base + '_dev'`。

- **前端**：`vite.config.ts` 读 `packages/app/package.json`，`define` 注入 `__MONOREPO_VERSION__` 与 `__IS_PROD__`（`mode === 'production'`）。`main.tsx` 启动 `console.info('tmex <display>')`。
- **gateway**：`apps/gateway/src/system/version.ts` 的 `getBaseVersion()` 分层取值：
  1. 构建期注入的 `TMEX_MONOREPO_VERSION`（`typeof` 守卫；`packages/app` `build:runtime` 与 `apps/gateway` `build` 均注入）——主生产/容器路径。
  2. production 兜底：`install-meta.json.cliVersion`。
  3. dev/test：仓库 `packages/app/package.json`。
  4. 兜底 `unknown`。
  - 后端在 `apps/gateway/src/index.ts`（dev）与 `packages/app/src/runtime/server.ts`（prod bundle）启动时输出版本。

构建期注入用 `bun build --define`：`packages/app/scripts/build-runtime.ts`、`apps/gateway/scripts/build.ts` 读版本并传 `--define TMEX_MONOREPO_VERSION="x.y.z"`；docker builder 增 `COPY packages/app/package.json`。

## 安装方式检测

`apps/gateway/src/system/install-info.ts`：

- 反推 installDir：优先 `TMEX_FE_DIST_DIR`（= `installDir/resources/fe-dist`）上溯两级，回退 `cwd`（服务 `WorkingDirectory=installDir`）。
- 读 `installDir/install-meta.json`：存在 → `installedViaCli=true`，`deployment` 由 `platform` 映射（darwin→launchd / linux→systemd / 其余→none）。
- 仅 production 才判 CLI 安装；dev/test 与 production-无-meta（docker/手动）→ 非 CLI。

`canSelfUpdate = isProd && installedViaCli && deployment !== 'none'`，同时满足需求：非 production 禁用、非 CLI 安装禁用。

## 检查更新与 changelog

`apps/gateway/src/system/update-check.ts`：

- 查 `https://registry.npmjs.org/tmex-cli`（`no-store`），取 `dist-tags.latest` 与 `time[latest]`，与当前版本 semver 比较得 `hasUpdate`。
- changelog：尽力从 `https://cdn.jsdelivr.net/npm/tmex-cli@<latest>/CHANGELOG.md` 拉取（`no-store`）；失败返回 null，前端回退展示「版本 + 发布时间」。

## 升级状态机与执行

`apps/gateway/src/system/upgrade.ts`，全局唯一 `UpgradeController`，状态 `idle / downloading / executing`：

1. `downloading`：临时目录 + 全新 `BUN_INSTALL_CACHE_DIR`（无视缓存）中 `bun add tmex-cli@<version>`（`process.execPath` 即 bun，必然可用）。此阶段 gateway 仍存活，失败回 `idle` 并经 `status()` 上报 error。
2. `executing`：`detached` 拉起下载包的 `bin/tmex.js upgrade --apply-current-package --install-dir <dir> --version <v>`（输出落 `installDir/upgrade.log`）。子进程停服务（杀掉本 gateway）→ 部署 → 重启；服务重启后新 gateway 启动即 `idle`。

依赖服务 unit 的 `KillMode=process`（systemd）/ `AbandonProcessGroup=true`（launchd），使 detached 子进程在服务进程被停时存活，完成自升级。

## API

`apps/gateway/src/api/system.ts`（挂在 `/api/system/`）：

- `GET /api/system/info` → `SystemInfo`
- `GET /api/system/update-check` → `UpdateCheckResult`
- `GET /api/system/upgrade` → `UpgradeStatus`（轮询）
- `POST /api/system/upgrade` `{version}` → 启动；非 `canSelfUpdate` 403、忙 409、缺版本 400

## 前端

设置页新增「版本与更新」Tab（`apps/fe/src/components/settings/version-tab.tsx`）：展示版本/安装方式/部署方式；「检查更新」拉取 latest + changelog（markdown 渲染）；`canSelfUpdate` 时「立即升级」弹窗警告（中断访问、可能影响 tmux 进程存活）→ `POST` 后轮询状态，跨服务重启检测完成并刷新版本；否则禁用并给出原因 + 终端升级提示。

## 验收

- dev：前端 console 与后端启动日志均为 `0.10.0_dev`；版本 Tab 升级禁用（原因：非 production）。
- production + CLI 安装（install-meta）：`/api/system/info` → `installedViaCli=true`、正确 deployment、`canSelfUpdate=true`。
- production 无 meta（docker）：`canSelfUpdate=false`。
- `update-check` 能查到 npm latest；状态机 `downloading→executing`，非法触发 403/409。

## 注意事项

- 升级会中断当前访问并重启服务，可能影响该服务托管的 tmux 进程存活——前端确认弹窗已明确告知。
- changelog 随包发布、仅含当前版本；历史版本（如 0.10.0 之前）无 CHANGELOG → CDN 404 → 前端回退版本+日期。
