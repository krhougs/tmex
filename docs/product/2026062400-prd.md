# tmex 产品需求文档（PRD）

> 版本：v0.13.0 · 快照日期：2026-06-24

---

## 1. 产品概述

### 1.1 定位

**tmex** 是一个面向 AI Agent 时代的自托管 tmux 终端工作区管理平台。它将 tmux 的多路复用能力通过 Web 界面暴露，允许用户从任意设备（电脑、平板、手机）通过浏览器管理本地和远程 SSH 主机上的终端会话，并集成了 AI Agent、屏幕监控（Watch）、多渠道通知等自动化能力。

### 1.2 目标用户

- 需要跨设备管理多台服务器终端的运维/开发工程师
- 使用 AI Coding Agent（Claude Code、Codex 等）进行开发并需要远程监控的开发者
- 需要长时运行任务（构建、训练、部署）并实时监控告警的团队

### 1.3 核心价值主张

| 价值点 | 描述 |
|--------|------|
| **跨设备终端访问** | 电脑、手机、平板打开浏览器即可继续工作，PWA 安装后体验接近原生 |
| **AI Agent 深度集成** | 服务端 Agent 绑定 tmux pane，可读屏、执行命令、发按键、搜索网页 |
| **自动化监控告警** | Watch 规则持续看屏，异常时通过 Telegram/微信/Webhook/浏览器推送 |
| **本地+远程统一管理** | 同一侧边栏同时管理本地和 SSH 远程设备 |
| **原生 tmux 兼容** | 基于 tmux Control Mode，可与 iTerm2 等原生客户端共用会话 |
| **Ghostty 终端内核** | 浏览器端终端由 Ghostty 官方 VT 内核（WASM）驱动，终端语义与原生一致 |

## 2. 系统架构

### 2.1 技术栈

| 层级 | 技术 |
|------|------|
| 运行时 | Bun.js（不兼容 Node.js） |
| 后端 | Bun HTTP/WebSocket Server（apps/gateway） |
| 前端 | React 19 + Vite + Tailwind CSS 4 + shadcn/ui（apps/fe） |
| 数据库 | SQLite（WAL 模式）+ Drizzle ORM |
| 终端模拟 | Ghostty VT 内核 → WebAssembly（packages/ghostty-terminal） |
| WebSocket 协议 | 自定义 Borsh 二进制序列化（packages/shared/ws-borsh） |
| 状态管理 | Zustand（前端）、内存 + SQLite（后端） |
| 构建发布 | npm（tmex-cli）、launchd/systemd 服务注册 |
| i18n | i18next，支持中英文 |

### 2.2 Monorepo 结构

```
tmex/
├── apps/
│   ├── gateway/          # 后端：HTTP API + WebSocket + 业务逻辑
│   └── fe/               # 前端：React SPA
├── packages/
│   ├── shared/           # 前后端共享类型、ws-borsh 编解码、i18n 资源
│   └── ghostty-terminal/ # Ghostty WASM 终端模拟器封装
├── scripts/              # 构建、开发、发布脚本
└── docs/                 # 技术文档
```

### 2.3 部署模型

- **单用户自托管**：通过 `npx tmex-cli init` 一键安装
- **服务管理**：macOS 用 launchd、Linux 用 systemd 注册为用户级服务
- **网络安全**：不内置鉴权，依赖 Cloudflare Access / Tailscale 等零信任网络保护
- **Docker**：支持 docker-compose 部署

## 3. 功能模块详述

### 3.1 设备管理（Device Management）

**目标**：统一管理本地和远程终端设备。

| 能力 | 描述 |
|------|------|
| 设备类型 | `local`（本地 tmux）、`ssh`（SSH 远程主机） |
| SSH 认证方式 | 密码、私钥、SSH Agent 转发、SSH Config 引用、自动探测 |
| 凭证安全 | 密码/私钥使用 AES-256-GCM 加密落库 |
| 设备树 | 侧边栏展示设备→窗口→面板树形结构，支持拖拽排序 |
| 连接测试 | 创建/编辑设备时可测试连接 |
| SSH 重连 | 断线自动重连，可配置最大重试次数和延迟 |
| 运行时状态 | 实时展示设备连接状态、tmux 可用性、最后在线时间、错误信息 |

**数据模型**：`devices`、`deviceRuntimeStatus`、`deviceTreeOrder`

### 3.2 终端子系统（Terminal）

**目标**：在浏览器中提供原生级终端体验。

#### 3.2.1 tmux Control Mode

- 通过 tmux control mode 协议与 tmux 服务端通信
- 实时接收 pane 输出、窗口生命周期事件、bell 通知
- 支持窗口创建/关闭/重命名、pane 关闭
- 与 iTerm2 等原生 tmux 客户端可共用同一 session

#### 3.2.2 Ghostty WASM 终端

- 使用 Ghostty 官方 VT 内核编译为 WebAssembly
- 完整终端语义解析，不依赖自研 ANSI 解析器
- 支持 alternate screen、滚动历史

#### 3.2.3 WebSocket 协议（ws-borsh）

- 自定义 Borsh 二进制序列化，高效传输终端数据
- 消息类型：状态快照、增量事件、终端 I/O、Agent 事件、Watch 事件
- Switch Barrier 机制确保 pane 切换时的流畅体验
- 分块传输大体积终端输出

#### 3.2.4 移动端适配

- 虚拟键盘不打乱终端布局
- 编辑器模式支持长命令编辑
- 可自定义终端快捷键栏（send 类发送控制序列、action 类触发前端行为）
- iOS Safari 特殊优化

#### 3.2.5 终端设置

- 字体选择/字号/行高（woff2 字体构建流水线）
- 自定义快捷键栏（全设备同步）
- 终端主题（明/暗）

### 3.3 AI Agent

**目标**：服务端 AI Agent 绑定 tmux pane，具备终端操作和网络搜索能力。

#### 3.3.1 会话模型

| 属性 | 描述 |
|------|------|
| 绑定 | 每个 Agent 会话绑定到一个设备的一个 pane |
| LLM Provider | 支持多 Provider（OpenAI Chat / OpenAI Responses 协议） |
| 模型管理 | 自动拉取可用模型列表 + 手动添加 + 逐个启停 |
| 写入模式 | `confirm`（默认，写操作需确认）/ `auto`（自动执行） |
| 消息队列 | 运行中可排队用户消息（step 边界注入或立即 steer） |
| 最大步数 | 每轮对话最大 tool call 步数限制 |

#### 3.3.2 工具能力

| 工具 | 描述 |
|------|------|
| `read_screen` | 读取当前绑定 pane 的终端屏幕内容 |
| `run_command` | 在 pane 中执行命令（headless Ghostty 模拟执行） |
| `send_keys` | 向 pane 发送按键序列 |
| `web_search` | 搜索网页（Tavily / Brave） |
| `fetch_url` | 抓取指定 URL 内容（SSRF 防护：拒绝回环/私网地址） |
| Provider Hosted Tools | 支持 Provider 原生工具（如 OpenAI image_generation） |

#### 3.3.3 安全机制

- 写操作（send_keys、run_command）默认需用户确认
- 凭证消毒：出站 LLM 请求通过 redaction middleware 消毒敏感信息
- 用户输入不改写只告警
- SSRF 防护

#### 3.3.4 实时交互

- WebSocket 流式推送：text delta、reasoning delta、tool call/result、确认请求
- 前端 Agent Panel：聊天线程、模型选择器、排队消息芯片

### 3.4 Watch 监控

**目标**：后台规则引擎持续监控终端屏幕，异常时主动告警。

#### 3.4.1 触发类型

| 类型 | 描述 |
|------|------|
| `match` | 正则匹配屏幕内容（支持分组提取） |
| `unchanged` | 屏幕内容超过指定时间未变化 |
| `llm` | LLM 根据自然语言条件判断屏幕内容 |

#### 3.4.2 规则配置

| 配置项 | 描述 |
|--------|------|
| 采样间隔 | 屏幕读取频率（秒） |
| 触发模式 | `once`（触发一次后停用）/ `repeat`（带冷却时间重复触发） |
| 冷却时间 | repeat 模式下两次触发间的最小间隔 |
| 无匹配行为 | `reset`（计数清零）/ `ignore`（忽略） |
| LLM 确认 | 正则匹配后用 LLM 二次确认 |
| LLM 摘要 | 触发时用 LLM 生成通知摘要 |
| AI 辅助建正则 | LLM 根据自然语言描述 + 屏幕样本生成正则表达式 |

#### 3.4.3 运行时状态

- 每条规则维护独立的状态机（最后采样时间、最后值、触发记录、连续错误计数）
- 内存 ring buffer 存储近期采样点

### 3.5 通知系统（Notification）

**目标**：多渠道推送终端事件通知。

#### 3.5.1 通知渠道

| 渠道 | 描述 |
|------|------|
| **浏览器推送** | Bell Toast + Notification Toast，支持独立启停 |
| **Telegram** | 多 Bot 支持，聊天审批授权机制，支持私聊/群组/频道 |
| **微信（iLink/ClawBot）** | 扫码登录，半主动推送语义（需 inbound context_token），8h 保活，单用户绑定 |
| **Webhook** | HMAC-SHA256 签名，可配置事件类型过滤 |

#### 3.5.2 事件类型

```
terminal_bell / terminal_notification
tmux_window_close / tmux_pane_close
device_tmux_missing / device_disconnect / session_created / session_closed
agent_confirmation_pending / agent_turn_finished / agent_error
watch_triggered / watch_model_unavailable / watch_rule_error
```

#### 3.5.3 通知检测

- 支持 BEL（`\a`）
- OSC 通知序列：OSC 9、OSC 99、OSC 777 `notify`、iTerm2 OSC 1337 `RequestAttention`
- 与 Claude Code、Codex、OpenCode 等主流 Agent 自动兼容

#### 3.5.4 节流

- Bell 通知节流（可配置秒数）
- Notification 通知节流（可配置秒数）
- 同类故障告警单次，恢复后重置

### 3.6 文件管理（Files）

**目标**：浏览和传输本地/远程设备上的文件。

| 能力 | 描述 |
|------|------|
| 文件浏览 | 设备绑定的白名单根目录（File Roots）下的目录树 |
| 文件查看 | 代码高亮、Markdown 预览、图片预览 |
| 文件传输 | rsync 统一传输（本地直接 rsync，SSH 走 rsync over SSH） |
| 分块上传 | init → chunk → commit 三阶段协议，NDJSON 流式进度 |
| 分块下载 | 带进度条和取消能力 |
| 上下文菜单 | 右键菜单操作（下载、打开等） |
| rsync 自动安装 | 远程设备缺少 rsync 时自动引导安装 |

**数据模型**：`fileRoots`（设备绑定的白名单根目录）

### 3.7 系统管理

#### 3.7.1 站点设置

- 站点名称/URL
- 语言切换（中/英）
- Bell/Notification 节流配置
- SSH 重连策略
- 各渠道通知开关

#### 3.7.2 自更新

- 设置页检查更新 + 一键升级
- CLI 命令升级：`npx tmex-cli upgrade`
- 升级状态机：`idle` → `downloading` → `executing`
- 失败自动回滚
- 显示目标版本 changelog

#### 3.7.3 CLI 工具（tmex-cli）

| 命令 | 描述 |
|------|------|
| `init` | 交互式安装（生成密钥、部署文件、注册服务） |
| `init --no-interactive` | 非交互安装（CI/自动化） |
| `upgrade` | 升级到最新版本 |
| `doctor` | 环境诊断 |
| `uninstall` | 卸载 |

#### 3.7.4 进程管理

- macOS：launchd KeepAlive + WatchPaths 重启策略
- Linux：systemd user service + 自动重启
- Gateway 支持优雅重启（运行时控制器）

## 7. 非功能需求

### 7.1 性能

- WebSocket 二进制协议（Borsh）减少序列化开销
- 终端输出分块传输
- Switch Barrier 避免 pane 切换时的闪烁
- 文件分块上传下载
- 通知节流防止风暴

### 7.2 可靠性

- SSH 自动重连（可配最大重试+延迟）
- Agent 会话/排队消息持久化（重启不丢）
- Watch 规则状态持久化
- 升级失败自动回滚
- Gateway 优雅重启

### 7.3 安全

- AES-256-GCM 凭证加密
- HMAC-SHA256 Webhook 签名
- Agent 写操作确认机制
- SSRF 防护（fetch_url 拒绝私网）
- 无内置鉴权，依赖网络层保护
- OSC passthrough 默认关闭（防终端逃逸）

### 7.4 兼容性

- 平台：macOS（launchd）、Linux（systemd）
- 浏览器：现代浏览器（Chrome/Firefox/Safari/Edge），PWA 支持
- 移动端：iOS Safari + Android Chrome，虚拟键盘优化
- tmux：原生 Control Mode 兼容
