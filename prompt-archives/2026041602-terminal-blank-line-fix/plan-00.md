## 背景

tmex 的终端在切换到某些 pane，或冷启动直接进入 pane 时，会出现多出一行非预期空白行、随后更新错位的问题。后续确认普通 shell 与 opencode 都能复现，因此问题更像是通用终端显示链路缺陷，而不是某个特定 TUI 的兼容问题。

当前分析结果显示，前端在把 history 和 live output 写入终端前，会统一执行 CRLF 归一化，这一步最可能改变真实输出语义。只有在该方向被排除后，才需要继续怀疑 ghostty-terminal 的 rows / viewport 映射。

## 注意事项

- 先测试，后实现。
- 只做最小修复，不扩大为重构。
- 优先修改 `apps/fe/src/components/terminal/Terminal.tsx`。
- 仅当主路径验证失败时，才继续修改 `packages/ghostty-terminal/src/terminal.ts`。
- 修复后必须执行手动可观测验证，而不只是静态检查。

## 任务清单

1. 为输出归一化逻辑补一个最小失败测试，覆盖跨 chunk 的 CR/LF 边界语义。
2. 运行该测试，确认按预期失败。
3. 在 `Terminal.tsx` 中做最小修改修复归一化状态传递或逻辑错误。
4. 重新运行针对性测试，确认通过。
5. 运行必要的类型/测试验证，并做一次手动场景验证。
6. 若问题仍存在，再进入 ghostty viewport 路径排查。

## 验收标准

- 新增测试能稳定覆盖本次回归场景。
- 修复后测试通过。
- 终端在普通 shell 场景下不再出现额外空白行或明显更新错位。
- 不引入新的 TypeScript 诊断错误。

## 风险评估

- 如果问题实际根因在 ghostty viewport 逻辑，仅修改归一化逻辑可能不足以彻底修复。
- 若测试过于抽象，可能无法准确覆盖真实 UI 症状，因此需要补充手动验证。
